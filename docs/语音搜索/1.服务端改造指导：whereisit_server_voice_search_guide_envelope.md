# WhereIsIt 服务端语音搜索改造指导

## 1. 文档目的

本文档用于指导 WhereIsIt 服务端增加一期语音搜索能力。目标是在不破坏现有文字搜索能力的前提下，新增：

1. 实时语音流接入
2. 第一阶段流式 STT
3. 第二阶段非流式最终识别
4. 最终文本搜索并返回现有物品列表结构

WhereIsIt 当前采用 FastAPI + SQLAlchemy + PostgreSQL 16 + Docker Compose。 README 已说明系统当前具备物品管理与查询能力，适合在现有 API 之上增加语音搜索接口。 citeturn781242view0turn781242view2

---

## 2. 服务端改造目标

一期服务端需要新增三类能力：

### 2.1 流式识别能力

职责：

- 接收 Android 客户端音频流
- 调用 sherpa-onnx streaming recognizer
- 持续返回 partial 文本

### 2.2 最终识别能力

职责：

- 接收一次完整语音段
- 调用 FunASR / Fun-ASR-Nano 非流式识别
- 产出最终文本

### 2.3 搜索编排能力

职责：

- 对最终文本做归一化
- 提取关键词
- 调用现有文字搜索逻辑
- 返回与文字搜索相兼容的结果结构


## 2.4 Envelope 返回规范（新增要求）

本次语音搜索相关 **HTTP 接口返回必须符合 WhereIsIt 现有 Envelope 规范**。建议在服务端 schema 层复用项目已有通用响应包装器，不为语音搜索单独发明一套裸 JSON 返回格式。

建议抽象结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "...": "业务数据"
  }
}
```

约束：

- 成功时：`code = 0`
- 失败时：返回项目既有错误码与错误消息
- 成功与失败都必须保持相同外层结构
- 本文档中后续提到的 `final_text`、`items`、`timing` 等字段，均默认位于 `data` 内
- WebSocket 消息不强制使用 Envelope；如果项目已有统一 WebSocket 消息壳，也可按现有约定扩展

## 3. 建议新增的模块结构

建议在后端新增一个 `voice_search` 模块，结构可以类似：

```text
app/
  api/
    routes/
      voice_search.py
  services/
    voice_search/
      session_manager.py
      streaming_asr_service.py
      offline_asr_service.py
      text_normalizer.py
      query_builder.py
      item_search_adapter.py
  schemas/
    voice_search.py
    common.py  # 复用或引用项目现有 Envelope schema
```

职责建议如下：

### `voice_search.py`
- 暴露 WebSocket 和 HTTP 接口

### `session_manager.py`
- 管理一次语音会话
- 保存会话状态、音频缓冲、开始时间、结束时间

### `streaming_asr_service.py`
- 封装 sherpa-onnx streaming recognizer
- 提供 `start_session()`、`feed_audio()`、`get_partial()`、`finish_stream()`

### `offline_asr_service.py`
- 封装 FunASR / Fun-ASR-Nano 非流式识别
- 提供 `recognize(audio_bytes)`

### `text_normalizer.py`
- 负责 query 归一化

### `query_builder.py`
- 从最终文本生成搜索输入

### `item_search_adapter.py`
- 调用现有物品搜索服务，而不是重写搜索主逻辑

## 4. 接口设计建议

## 4.1 WebSocket：实时语音流

建议：

- `WS /api/voice-search/stream`

### 连接建立后流程

1. 客户端发送 `start`
2. 服务端创建 session
3. 客户端持续发送 `audio`
4. 服务端持续返回 `partial`
5. 客户端发送 `stop`
6. 服务端结束第一阶段并回 `finalizing`

### 建议消息格式

#### 客户端 -> 服务端

```json
{ "type": "start", "sample_rate": 16000, "channels": 1, "encoding": "pcm_s16le" }
```

```json
{ "type": "audio", "seq": 1, "data": "<base64>" }
```

```json
{ "type": "stop" }
```

#### 服务端 -> 客户端

```json
{ "type": "session", "session_id": "vs_20260421_xxx" }
```

```json
{ "type": "partial", "text": "小米" }
```

```json
{ "type": "partial", "text": "小米 note" }
```

```json
{ "type": "finalizing", "session_id": "vs_20260421_xxx" }
```

> 是否直接在 WebSocket 中返回最终搜索结果，可以视实现复杂度决定。一期更推荐 WebSocket 只负责实时阶段，最终识别与搜索改用 HTTP。

## 4.2 HTTP：最终识别 + 搜索

建议新增：

- `POST /api/voice-search/finalize`

### 方案 A：上传 session_id

如果 WebSocket 会话内已缓存整段音频：

请求：

```json
{
  "session_id": "vs_20260421_xxx",
  "first_stage_text": "小米 note 手机"
}
```

### 方案 B：直接传音频文件

如果不希望 WebSocket 侧缓存整段音频太久，则可在松开时直接用 HTTP 上传：

- multipart file
- optional `first_stage_text`

### 响应结构

建议结构与现有文字搜索接口一样，统一走 Envelope：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "session_id": "vs_20260421_xxx",
    "final_text": "红米note8手机",
    "normalized_query": "红米 note8 手机",
    "keywords": ["红米", "note8", "手机"],
    "items": [
      {
        "id": 123,
        "name": "note手机",
        "brand": "小米",
        "location": "书房抽屉"
      }
    ],
    "timing": {
      "offline_asr_ms": 420,
      "search_ms": 32
    }
  }
}
```

失败示例：

```json
{
  "code": 4001001,
  "message": "voice search session not found",
  "data": null
}
```

## 5. sherpa-onnx 流式服务接入建议

sherpa-onnx 官方支持 streaming STT、Android、WebSocket server/client 和 VAD/endpoint detection，适合用作第一阶段实时回显引擎。 citeturn966714view3turn267426search1turn267426search16

### 5.1 推荐接入方式

服务端内部不要在每个请求里临时初始化模型，而是：

- 应用启动时初始化 streaming recognizer
- 每个连接维护独立 stream/session
- 每个会话持有：
  - recognizer stream
  - audio buffer
  - partial text cache

### 5.2 音频参数

- sample rate: 16000
- channels: 1
- sample format: signed 16-bit PCM

### 5.3 实时回显频率

建议：

- 每收到 N 帧或每 100~200ms 刷新一次 partial
- 避免每一帧都向客户端推文本，减少 UI 抖动与网络开销

### 5.4 一期热词建议

把现有数据库中的：

- 品牌名
- 常见物品名
- 常见类别

提取为热词列表，供 sherpa-onnx 第一阶段使用。

这一步不要过度复杂，初版只要能提升常见词识别即可。

---

## 6. FunASR / Fun-ASR-Nano 非流式接入建议

FunASR 官方仓库说明其提供 ASR、VAD、文本后处理等能力；Fun-ASR 路线强调中文、多方言、多区域口音覆盖，并持续维护实时与离线转写服务。 citeturn966714view2turn966714view4

### 6.1 一期推荐接入方式

在服务端单独封装 `offline_asr_service`：

```python
class OfflineAsrService:
    def recognize(self, audio_bytes: bytes) -> str:
        ...
```

### 6.2 建议实现原则

1. 模型常驻内存
2. 不要每次请求动态加载模型
3. 只支持短音频段识别
4. 对识别失败要有清晰错误码

### 6.3 输入输出

输入：
- 一段完整音频

输出：
- `final_text`
- optional 置信度
- optional tokens

一期只要求拿到 `final_text` 即可。

## 7. 文本归一化模块设计

建议新增 `text_normalizer.py`，至少实现以下规则：

### 7.1 基础规则

- 去首尾空白
- 连续空格压缩
- 英文转小写
- 中文标点清理
- 全角半角统一

### 7.2 搜索友好规则

- 去掉常见语气词：
  - 帮我找
  - 找一下
  - 我想找
  - 那个
- 型号数字合并：
  - `note 8` -> `note8`
  - `mate 40` -> `mate40`
- 可选：中英混合间距归一

### 7.3 实现建议

提供两个输出：

```python
@dataclass
class NormalizedText:
    raw_text: str
    display_text: str
    query_text: str
```

含义：

- `raw_text`：原始识别结果
- `display_text`：给前端展示
- `query_text`：用于检索

## 8. 查询构造与搜索复用

WhereIsIt 已有物品查询能力，因此建议不要为语音搜索重新写一套 SQL，而是做一个适配层，把语音最终文本映射成现有文字搜索输入。 citeturn781242view0

### 8.1 `query_builder.py` 职责

输入：
- `final_text`
- `normalized_query`

输出：
- `query_variants`
- `keywords`

### 8.2 一期的 query variants 建议

例如最终文本为：

- `帮我找红米 note 8 手机`

归一化后可以构造：

1. `红米note8手机`
2. `红米 note8 手机`
3. `note8 手机`
4. `红米 手机`

### 8.3 搜索复用策略

建议：

1. 先调用现有文字搜索接口或服务方法
2. 使用 `query_variants` 多次查询
3. 合并结果并去重
4. 简单排序

### 8.4 一期排序建议

- 名称完整包含优先
- 品牌命中优先
- 名称部分匹配其次
- 结果去重后按分数降序

## 9. 会话管理与音频缓存

### 9.1 session 需要保存的内容

建议保存：

- `session_id`
- `created_at`
- `client_id/user_id`
- `audio_chunks`
- `partial_text`
- `first_stage_final_text`
- `state`

### 9.2 生命周期

状态可以设计为：

- `INIT`
- `STREAMING`
- `STOPPED`
- `FINALIZING`
- `DONE`
- `FAILED`

### 9.3 清理策略

- `DONE` 后保留短时间（如 1~5 分钟）便于重试
- 超时未完成的会话自动清理
- 音频缓存要限制最大长度

## 10. 错误处理建议

### 10.1 WebSocket 常见错误

- 音频格式不支持
- 未先 start 就发 audio
- session 超时
- 模型不可用

### 10.2 finalize 常见错误

- session 不存在
- 无音频数据
- 第二阶段识别失败
- 搜索异常

### 10.3 推荐错误码

- `VOICE_SESSION_NOT_FOUND`
- `VOICE_AUDIO_EMPTY`
- `VOICE_STREAMING_ASR_ERROR`
- `VOICE_OFFLINE_ASR_ERROR`
- `VOICE_SEARCH_ERROR`

## 12. Docker / 部署建议

WhereIsIt 使用 Docker Compose 部署。新增语音搜索后，建议有两种部署方式：

### 方案 A：全部在同一个 FastAPI 容器内

优点：
- 最简单

缺点：
- 模型依赖和 API 进程耦合较重

### 方案 B：拆成独立语音服务容器

推荐：

- `whereisit-api`
- `whereisit-voice`

其中：

- `whereisit-voice` 负责 sherpa-onnx 与 FunASR
- `whereisit-api` 通过内部 HTTP 或 Python service adapter 调用

优点：
- 模型生命周期独立
- 后续更好扩展
- 更利于单独调优

一期如果希望改动最小，可以先选 A；如果从维护角度考虑，更推荐 B。

## 13. 日志与指标

至少埋以下字段：

- `session_id`
- `user_id`
- `audio_duration_ms`
- `first_stage_text`
- `final_text`
- `normalized_query`
- `keyword_count`
- `search_result_count`
- `streaming_latency_ms`
- `offline_asr_latency_ms`
- `search_latency_ms`
- `clicked_item_id`（如果后续客户端上报）

### 13.1 为什么这些日志重要

后续判断：

- 第一阶段回显是否足够稳定
- 第二阶段是否明显优于第一阶段
- 哪些词最容易识别错
- 哪些搜索文本召回差

## 14. 实施步骤建议

### 步骤 1：加 WebSocket 基础通路
- 新建 `voice_search.py`
- 接收 `start/audio/stop`
- 先不接真正模型，只做假回显打通

### 步骤 2：接入 sherpa-onnx
- 完成 partial 文本回传
- 完成音频缓存

### 步骤 3：接入 finalize 接口
- 完成 session_id -> offline ASR -> final_text

### 步骤 4：接入现有搜索
- final_text -> normalizer -> query builder -> item search

### 步骤 5：补监控与错误处理
- 日志、超时、清理策略

---

## 15. 服务端验收标准

满足以下条件即可认为一期服务端可用：

1. Android 可通过 WebSocket 持续上传音频
2. 服务端可持续返回 partial 文本
3. 用户松开后，服务端能在合理时间内返回 `final_text`
4. 服务端能把 `final_text` 转成搜索结果
5. 返回结构能直接被 Android 现有结果页消费

---

## 16. 最终建议

服务端改造的关键不是“做一套全新的语音业务系统”，而是：

- 用 `sherpa-onnx` 做实时识别入口
- 用 `FunASR / Fun-ASR-Nano` 做最终识别
- 用适配层复用现有搜索逻辑

这样能以最小改动，把 WhereIsIt 现有 FastAPI + PostgreSQL 体系扩展为一套可上线的一期语音搜索方案。
