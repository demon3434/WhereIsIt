# WhereIsIt 语音搜索：服务端开发与调测顺序

本文面向 **WhereIsIt 服务端开发人员**，目标是在尽量少返工的前提下，完成语音搜索一期能力，并为 Android 客户端提供稳定的联调接口。

---

## 1. 开发目标

一期服务端需要提供两类能力：

1. **非流式最终搜索接口**
   - 接收客户端上传的完整语音
   - 使用 FunASR-Nano 做非流式识别
   - 对识别文本做轻量规范化
   - 复用现有文字搜索逻辑
   - 以 Envelope 规范返回结果

2. **流式实时字幕接口**
   - 接收客户端实时上传的音频帧
   - 使用 sherpa-onnx 做实时识别
   - 把实时文本片段推送给客户端
   - 该接口主要用于“实时显示识别文字”，不承担最终搜索结果输出

---

## 2. 推荐开发顺序

### 阶段 A：先打通“非流式最终搜索”闭环

这是最优先开发的部分，因为它决定语音搜索是否真正可用。

#### A1. 定义接口契约
先冻结接口文档，至少明确：

- 路径：`POST /api/voice-search/final`
- 请求方式：`multipart/form-data`
- 请求字段：
  - `audio`：语音文件
  - `audio_format`：如 `wav` / `pcm`
  - `sample_rate`：如 `16000`
  - `channels`：如 `1`
  - `language`：一期默认 `zh`
- 响应 Envelope：
  - `code`
  - `message`
  - `data`

建议 `data` 至少包含：

```json
{
  "recognizedText": "红米note8手机",
  "normalizedText": "红米note8手机",
  "items": [],
  "debug": {
    "engine": "funasr-nano",
    "elapsedMs": 320
  }
}
```

#### A2. 先不接真实模型，先做 mock
在真实 ASR 接入前，先让接口返回固定识别文本，例如：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "recognizedText": "小米手机",
    "normalizedText": "小米手机",
    "items": [ ...现有搜索结果... ]
  }
}
```

这样可以让 Android 客户端提前开始联调，不必等待模型部署完成。

#### A3. 接入现有文字搜索逻辑
把语音接口的最终检索统一复用到现有文字查询服务，不要单独复制一套搜索逻辑。

服务端内部建议链路：

1. 上传音频
2. 语音转文本
3. 文本规范化
4. 调用现有物品搜索 service
5. 包装为 Envelope 返回

#### A4. 接入 FunASR-Nano
在 mock 跑通后再接真实模型：

- 新建独立 `speech_service` 或 `stt_service`
- 负责：
  - 音频格式校验
  - 必要时转码到 16k / mono
  - 调用 FunASR-Nano
  - 返回文本

#### A5. 增加文本规范化
一期只做轻量规则，不做复杂字典：

- 去掉语气词：`帮我找一下`、`给我找`、`那个`
- 数字统一：`八 -> 8`
- 空格归并：`note 8 -> note8`
- 英文大小写统一

#### A6. 加日志与调试信息
建议记录：

- 请求时间
- 音频长度
- ASR 原始输出
- 规范化文本
- 最终查询词
- 查询耗时
- 返回结果数量

---

### 阶段 B：再开发流式实时识别接口

在非流式闭环跑通后，再上流式。

#### B1. 定义 WebSocket 协议
建议路径：`GET /ws/voice-search/stream`

客户端行为：
- 建立连接
- 按住期间持续发送音频帧
- 松开后发送结束消息

服务端行为：
- 接收 PCM 帧
- 输入 sherpa-onnx streaming recognizer
- 周期性推送实时文本

建议消息类型：

客户端发：

```json
{ "type": "start", "sampleRate": 16000, "channels": 1 }
{ "type": "audio", "seq": 1, "payload": "base64..." }
{ "type": "audio", "seq": 2, "payload": "base64..." }
{ "type": "stop" }
```

服务端回：

```json
{ "type": "partial", "text": "红米" }
{ "type": "partial", "text": "红米note" }
{ "type": "partial", "text": "红米note8" }
{ "type": "final", "text": "红米note8手机" }
```

注意：WebSocket 消息不强制套 Envelope；Envelope 只要求 HTTP 接口统一。

#### B2. 先用伪流式回包调通
在接 sherpa-onnx 前，可先写一个假实现：

- 收到第 3 包返回 `红米`
- 收到第 6 包返回 `红米note`
- 收到 stop 返回 `红米note8手机`

这样 Android 可以先完成页面实时字幕交互。

#### B3. 接入 sherpa-onnx
真实接入后再调：

- 帧长
- 缓冲大小
- partial 推送频率
- stop 后 final 输出时机

---

### 阶段 C：联调与稳定性优化

#### C1. 统一错误码
建议约定：

- `0`：成功
- `4001`：音频格式不支持
- `4002`：音频为空
- `5001`：STT 执行失败
- `5002`：搜索执行失败

#### C2. 控制超时
一期建议：

- 非流式最终搜索接口超时：5~10 秒
- 流式连接空闲超时：30 秒

#### C3. 增加 mock 开关
建议通过配置切换：

- `VOICE_SEARCH_MOCK_FINAL=true`
- `VOICE_SEARCH_MOCK_STREAM=true`

便于联调和排障。

---

## 3. 推荐调测顺序

### 3.1 单元调试
先独立验证：

1. 音频上传与解析
2. FunASR-Nano 本地调用
3. sherpa-onnx 流式识别
4. 文本规范化函数
5. 复用现有文字搜索 service

### 3.2 接口联调
顺序建议：

1. `/api/voice-search/final` mock
2. `/api/voice-search/final` 真实模型
3. `/ws/voice-search/stream` mock
4. `/ws/voice-search/stream` 真实模型

### 3.3 真机调测
联调时重点观察：

- 上传音频采样率是否一致
- 非流式识别耗时
- partial 是否抖动过多
- stop 后 final 文本是否及时
- 搜索结果是否复用现有文字查询展示数据结构

---

## 4. 服务端开发注意事项

### 不要先做的事情
一期先不要做：

- 大而全的品牌/型号字典系统
- 复杂二阶段纠错流水线
- 独立的语音搜索结果 DTO 体系
- 与文字搜索完全分离的新搜索服务

### 一期应坚持的原则

- 先复用现有文字搜索
- 先保证闭环可用
- 先让客户端能联调
- 先让接口稳定，再谈识别优化

---

## 5. 交付检查清单

服务端一期完成时，至少应满足：

- [ ] `POST /api/voice-search/final` 可用
- [ ] 返回 Envelope
- [ ] 能接收 Android 上传音频
- [ ] 能完成非流式识别
- [ ] 能复用现有文字搜索
- [ ] `GET /ws/voice-search/stream` 可联调
- [ ] 能返回 partial / final 文本
- [ ] 有最基本日志与错误码
- [ ] 支持 mock 模式

---

## 6. 与 Android 的协作结论

服务端 **不需要全部开发完**，Android 才能开始。

推荐协作方式：

1. **先冻结接口契约**
2. **服务端先提供 mock 接口**
3. **Android 先接 mock 开发页面和交互**
4. **服务端再替换真实 ASR 实现**
5. **最后一起做真机联调**

也就是说：

- 可以并行开发
- 不建议串行等待
- 但必须先统一协议、字段和返回结构
